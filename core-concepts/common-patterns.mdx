## Base URL

All DataHub API endpoints follow a consistent URL structure:

```
https://api.datahub.syncspider.com/api/v1/apps/{hub-slug}/
```

The default API domain is `api.datahub.syncspider.com`. Replace `{hub-slug}` with the kebab-case hub identifier.

| Hub | Slug |
|-----|------|
| AttributesHub | `attributes-hub` |
| AuthHub | `auth-hub` |
| CustomerHub | `customer-hub` |
| EventHub | `event-hub` |
| LicenseHub | `license-hub` |
| MediaHub | `media-hub` |
| NotificationHub | `notification-hub` |
| OrderHub | `order-hub` |
| ProductHub | `product-hub` |
| RuleHub | `rule-hub` |
| SettingsHub | `settings-hub` |
| TaxHub | `tax-hub` |

## Path Notation in this Documentation

Documentation uses two styles depending on context — and they always mean the same thing:

| Style | Where you see it | Example |
|---|---|---|
| **Absolute URL** | curl snippets, HTTP request samples, runnable code | `https://api.datahub.syncspider.com/api/v1/apps/order-hub/carts` |
| **Relative path** | endpoint tables on a hub page, prose like "call `GET /products`" | `GET /carts` (on the OrderHub page) |

Relative paths in a hub page's tables are **always relative to that hub's base URL** declared at the top of the page. Two endpoints written as `GET /carts` on the OrderHub page and `GET /products` on the ProductHub page resolve to:

```
https://api.datahub.syncspider.com/api/v1/apps/order-hub/carts
https://api.datahub.syncspider.com/api/v1/apps/product-hub/products
```

## Header Naming vs JSON Field Naming

DataHub follows standard HTTP and JSON conventions, which means the **same value** appears with different casing depending on where it lives:

| Location | Convention | Example |
|---|---|---|
| HTTP request/response header | **kebab-case** | `session-uuid: 35cfd665-...` |
| JSON request/response body field | **snake_case** | `"session_uuid": "35cfd665-..."` |

The most common pair is `session-uuid` (header — for guest carts) ↔ `session_uuid` (body — the cart's stored field). They identify the **same** guest session; the casing difference is purely the HTTP-vs-JSON convention. Other affected pairs include `language-code` ↔ `language_code` and `installation-key` ↔ `installation_key` (rarely seen in bodies).

## Pagination

All list (index) endpoints return a flat DataHub pagination envelope:

```json
{
  "data": [ { "id": 1, "name": "..." }, ... ],
  "current_page": 1,
  "total_pages": 12,
  "per_page": 25,
  "total": 287
}
```

| Field | Meaning |
|---|---|
| `data` | The page of items. |
| `current_page` | 1-based. |
| `total_pages` | Number of pages with the currently applied `per_page`. |
| `per_page` | The page size that was actually applied. |
| `total` | Total matching rows after filters, before pagination. |

There is **no** `meta`, `links`, `from`, or `to` — this envelope is DataHub-flavored, not stock Laravel.

Control pagination with:

| Parameter | Default | Notes |
|---|---|---|
| `page` | `1` | Page number, 1-based. Walk by incrementing until `current_page == total_pages`. |
| `per_page` | `25` | Page size. **Snake-case** is the supported name. |

<Callout kind="warning" title="Use per_page (snake_case)">
The camelCase variant `perPage` is **silently ignored** — the request falls back to the default 25. Always send `per_page`.
</Callout>

### Walking Pages Efficiently

For bulk reads, build the URL yourself and increment `page`:

```js
let page = 1;
while (true) {
  const res = await api.get(
    '/products' +
    '?filter[status]=[{"condition":"=","value":"public"}]' +
    '&include=mainMedia' +
    '&per_page=200' +
    `&page=${page}`
  );
  for (const product of res.data) process(product);
  if (page >= res.total_pages) break;
  page += 1;
}
```

### Cursor-style Sync

When you need to incrementally pull a changing dataset (catalog sync, order replication), do not paginate by `page` — concurrent inserts and updates can shift rows between pages. Use a sort + filter on `_updated_at` instead:

```
?sort=_updated_at
&filter[_updated_at]=[{"condition":">","value":"2024-09-01T10:00:00Z"}]
&per_page=200
```

Store the highest `_updated_at` you saw and use it as the next cursor.

### Total Counts

`total` is the count after filters, before pagination. It is expensive on large collections — request it once and cache the result if the UI needs a "Showing X of Y" label. Every page also returns `total` but recomputing it is the same DB cost each time.

## Sorting

Sort results using the `sort` query parameter. Prefix a field name with `-` for descending order:

```
?sort=-_created_at,name
```

This sorts by `_created_at` descending, then by `name` ascending.

<Callout kind="tip" title="Common Sort Fields">
Most entities support sorting by `_created_at`, `_updated_at`, `name`, and `id`. Check each hub's documentation for entity-specific sort fields.
</Callout>

## Response Formats

### Success (Single Resource)

```json
{
  "data": {
    "id": 1,
    "name": "Example Resource",
    "_created_at": "2024-01-15T10:30:00Z",
    "_updated_at": "2024-01-15T10:30:00Z"
  }
}
```

### Success (Collection with Pagination)

```json
{
  "data": [
    { "id": 1, "name": "First" },
    { "id": 2, "name": "Second" }
  ],
  "current_page": 1,
  "total_pages": 10,
  "per_page": 25,
  "total": 250
}
```

### Error Response

```json
{
  "message": "The given data was invalid.",
  "errors": {
    "field_name": ["Error description."]
  }
}
```

For a complete list of error codes, see [Error Handling](/core-concepts/error-handling).

## HTTP Methods

| Method | Usage |
|--------|-------|
| `GET` | Retrieve resources (list or single) |
| `POST` | Create new resources |
| `PUT` | Update existing resources (full replacement) |
| `PATCH` | Partial update or upsert operations |
| `DELETE` | Remove resources |

## Required Headers

All authenticated requests must include:

```
Authorization: Bearer {api_token}
Accept: application/json
```

For requests with a body (POST, PUT, PATCH), also include:

```
Content-Type: application/json
```
