## Two Modes, One API

DataHub supports two distinct authentication modes. Pick the one that matches your use case:

| Use case | Required headers | Where it runs |
|---|---|---|
| **Storefront** (browse catalog, anonymous) | `installation-key` (+ `session-uuid` if cart) | Browser, mobile app, public-facing |
| **Storefront with logged-in B2B customer** | `installation-key` + `x-auth-token` (+ `session-uuid`) | Browser, mobile app, after `/auth-hub/login` |
| **Server-to-server** (admin, ERP sync, integrator) | `Authorization: Bearer {api_token}` | Backend only — never in a browser |
| **Public webhook / payment callback** | none (URL contains `{installationKey}`) | Provider callback (Stripe, PayPal, etc.) |

<Callout kind="warning" title="Bearer token must stay on the server">
The `Authorization: Bearer` token is a tenant-level secret. **Do not** ship it to a browser, mobile app, or any other untrusted client. Storefronts use `installation-key` (safe to expose) plus an optional per-customer `x-auth-token` (issued at login).
</Callout>

## What Each Endpoint Class Needs

The headers you must send depend on what the endpoint *does* — not just where you call from. Use this matrix as a quick check before issuing a request.

| Endpoint class | Examples | `installation-key` | `x-auth-token` | `session-uuid` | Bearer | Notes |
|---|---|:-:|:-:|:-:|:-:|---|
| Public catalog read | `GET /product-hub/products`, `GET /product-hub/categories`, `GET /settings-hub/locales` | ✅ | — | — | — | Anyone with the installation key can read public catalog data. |
| Storefront login | `POST /auth-hub/login`, `POST /auth-hub/register` | ✅ | — | optional | — | Returns the `x-auth-token` you use on subsequent calls. |
| Guest cart / checkout | `PATCH /order-hub/carts`, `POST /order-hub/checkout/preview` | ✅ | — | ✅ | — | Cart is keyed by `session-uuid` until the customer logs in. |
| Logged-in customer | `GET /auth-hub/me`, `GET /customer-hub/customers/{id}`, `GET /order-hub/orders` | ✅ | ✅ | optional | — | Customer identity comes from `x-auth-token`. |
| Admin / ERP / integrator | All write operations on tenant-wide resources (`POST /product-hub/products`, bulk imports, settings writes) | — | — | — | ✅ | A single `Authorization: Bearer` token replaces installation-key + x-auth-token. |
| Inbound webhook (provider → DataHub) | `POST /order-hub/payment/webhook/{installationKey}/{provider}/{order_id}` | — | — | — | — | No headers needed — the installation key is in the URL. The provider signs the body. |

A few rules implied by the matrix:

- The `installation-key` header alone is enough for **reads** of public storefront data, but **never** for writes against tenant-wide resources.
- The `x-auth-token` header **requires** `installation-key` too — the two are checked together. A token without an installation key returns `401`.
- `Authorization: Bearer` **replaces** the storefront pair. Do not send both — the API picks one and ignores the other, which makes 401s confusing.
- On the auth-hub itself, **don't** send a Bearer token. AuthHub endpoints are designed for the storefront flow; mixing in a server-to-server token will return `401`.

## Headers in Detail

### `installation-key` (storefront, required)

Identifies the **tenant installation** (which DataHub shop the request is for). A UUID issued when the shop was provisioned. Safe to expose in browser code.

Without this header, storefront requests return `401 Unauthenticated`.

Learn more in [Installation Key](/authentication/installation-key).

### `x-auth-token` (storefront, optional)

Identifies the **logged-in B2B customer**. Issued by `POST /api/v1/apps/auth-hub/login` and revoked by `POST /api/v1/apps/auth-hub/logout`.

Include it on requests that need the customer's identity (cart, checkout, account, orders). Omit it for guest browsing.

Tokens use a **sliding idle window**: each authenticated request refreshes the token's last-used timestamp. A token that goes unused for too long is rejected and the customer must log in again.

Learn more in [Storefront Customer Auth](/authentication/storefront-customer).

### `Authorization: Bearer {api_token}` (server-to-server, required)

Identifies an **integrator or admin** acting on behalf of the tenant. Issued through the DataHub admin panel.

Use this only from a trusted backend. Never expose it to end-users.

Learn more in [Server-to-Server Tokens](/authentication/server-to-server).

### `session-uuid` (storefront, recommended)

Identifies a **guest cart / browsing session** that survives login. The storefront generates a UUID on first visit, stores it in `localStorage`, and sends it on every request. The API echoes it back on the response so the client can pick up server-side rotations.

Required when the storefront supports guest checkout — the cart is keyed by this UUID until the customer logs in, at which point the guest cart is merged onto the customer account. After login it can still be sent; it is ignored where it is not relevant.

<Callout kind="info" title="Header vs body field naming">
This value travels as the kebab-case header `session-uuid` but appears inside JSON bodies and responses as the snake_case field `session_uuid` — same value, two casings to follow standard HTTP and JSON conventions. See [Header Naming vs JSON Field Naming](/core-concepts/common-patterns#header-naming-vs-json-field-naming) for the full convention.
</Callout>

### `language-code` (storefront, optional)

Two-letter or BCP-47 locale (e.g. `en-US`, `de-AT`, `sr-RS`) used to select localized strings — product names, descriptions, notification templates, attribute values. Falls back to the installation's default locale when missing.

### `Accept: application/json` (always)

Required on every request. Without it the API may return HTML for some error paths.

### `Content-Type: application/json` (POST/PUT/PATCH)

Required whenever the request has a body.

## Full Storefront Request Example

A typical request from a logged-in customer browsing a shop in German looks like:

```http
GET /api/v1/apps/order-hub/carts HTTP/1.1
Host: api.datahub.syncspider.com
Accept: application/json
Content-Type: application/json
installation-key: 7e26723d-6b84-493d-8f62-7add54b94470
x-auth-token: 1.JDJ5JDEyJDR5VGJEQThxcUdROVNQQ1Rn...
session-uuid: 35cfd665-064a-48b0-b411-03a4f7af44d6
language-code: de-AT
Origin: https://shop.example.com
```

Server-to-server callers replace `installation-key` + `x-auth-token` with a single `Authorization: Bearer {api_token}` header and typically omit `session-uuid` and `Origin`.

## Error Responses

### 401 Unauthorized

Returned when authentication is missing or invalid.

```json
{ "message": "Unauthenticated." }
```

On `GET /api/v1/apps/auth-hub/me` the response is more specific:

```json
{
  "error": "Unauthorized",
  "messages": ["No x-auth-token header was provided."],
  "code": 401
}
```

Common causes:
- Missing `installation-key` on a storefront request.
- Missing `Authorization` header on a server-to-server request.
- `x-auth-token` expired (no activity within the idle window) or revoked (logout).
- Token copy/paste with extra whitespace or line breaks.

### 403 Forbidden

The credentials are valid but the caller is not allowed to perform this action.

The most common case is a **shop customer trying to reach an admin-only endpoint** (back-office hubs such as SettingsHub, LicenseHub, NotificationHub admin routes). If `x-auth-token` resolves a shop customer, those endpoints reject the request with 403 even though `installation-key` is valid.

Other cases:
- Sales-agent-only endpoints (`/auth-hub/impersonate/*`) called by a non-agent.
- Server-to-server token with insufficient scope (e.g. read-only token attempting a write).

## CORS

The API echoes the request `Origin` header back and supports credentials, which means storefront calls from any origin currently succeed. See [CORS and Browser Clients](/authentication/cors-and-browsers) for the current behavior and recommended deployment patterns.
