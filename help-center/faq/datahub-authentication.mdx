## Which header do I use?

It depends on where your code runs.

| You are building… | Send these headers |
|---|---|
| A storefront (browser, mobile app) — guest browsing | `installation-key` |
| A storefront — logged-in B2B customer | `installation-key` + `x-auth-token` |
| A backend integration (ERP, sync job, BFF, MCP server) | `Authorization: Bearer {api_token}` |

The Bearer token must **never** be shipped to a browser. See the [Authentication overview](/authentication) for the full matrix.

## Where do I get the installation key?

From the DataHub admin panel, on the installation detail page. Your DataHub administrator provides it during onboarding. The installation key is safe to include in browser code. See [Installation Key](/authentication/installation-key).

## Where do I get an API token (Bearer)?

API tokens are issued through the DataHub admin panel. Contact your administrator if you do not have access. See [Server-to-Server Tokens](/authentication/server-to-server).

## How does a customer get an `x-auth-token`?

Call `POST /api/v1/apps/auth-hub/login` with the customer's email and password (and `installation-key`). The response contains a `token` value — that is the `x-auth-token` for subsequent requests. See [Storefront Customer Auth](/authentication/storefront-customer).

## Does one credential work across all hubs?

Yes. A single `installation-key` (and, when logged in, `x-auth-token`) is valid against every storefront-facing hub. A single Bearer token is valid against every back-office hub. The hub-specific URL prefix (`/api/v1/apps/{hub-slug}/`) only chooses *which* endpoints you can reach — it does not require a hub-specific credential.

## Are there any unauthenticated endpoints?

A few:

- `GET /api/v1/apps/media-hub/cdn/{domain}/{path}` — public CDN proxy for media.
- `POST /api/v1/apps/auth-hub/{installationKey}/reset-password` — completed from an email link.
- `POST /api/v1/apps/auth-hub/{installationKey}/verify-email` — completed from an email link.
- `GET /api/v1/apps/order-hub/payment/webhook/{installationKey}/{provider}` — payment provider callback.
- `GET /api/v1/apps/order-hub/order/payment/{installationKey}/{order_id}` — payment provider redirect.

The installation key in the URL path replaces the header for these.

## What happens when my customer's token expires?

Customer tokens use a **sliding idle window** — every authenticated request refreshes the token's last-used timestamp. If the customer goes inactive for the configured window, the next request returns `401`. The customer must log in again to get a fresh token.

There is no explicit refresh endpoint today.

## Can I mix Bearer with `installation-key`?

No. When a Bearer token is present the API uses it and ignores the storefront headers. Pick one mode per request.

## I get `403`. What does it mean?

The credentials are valid but the action is not allowed. Most common causes:

- A logged-in shop customer (`x-auth-token` present) called a back-office endpoint (admin-only hub or admin-only route).
- A read-only Bearer token attempted a write.
- A non-agent customer called an `/impersonate/*` endpoint.

See [Authentication Errors](/help-center/troubleshooting/authentication-errors) for full troubleshooting.

## I get `401`. What does it mean?

Authentication is missing or invalid:

- `installation-key` missing, malformed, or unknown.
- `Authorization` header missing on a server-to-server call.
- `x-auth-token` expired (idle too long) or revoked (logout).
- Extra whitespace, line breaks, or URL encoding in the token.

See [Authentication Errors](/help-center/troubleshooting/authentication-errors).
